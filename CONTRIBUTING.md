# Contributing Guide

Thanks for contributing to the Ghost Thermostat BuilderOps Pack.

## Principles

- Favor clarity and reproducibility over cleverness.
- Keep docs actionable and field-ready.
- Document intent and operational impact.

## Workflow

1. Create or reference an issue.
2. Create a branch from `main`.
3. Make focused changes.
4. Open a pull request with test/validation notes.
5. Address review feedback and merge.

## Branch Naming

Use one of:

- `feat/<short-description>`
- `fix/<short-description>`
- `docs/<short-description>`
- `chore/<short-description>`

## Pull Request Checklist

- [ ] Change has a clear purpose tied to an issue or operational need.
- [ ] Documentation is updated where relevant.
- [ ] `CHANGELOG.md` is updated for user-visible changes.
- [ ] Content is consistent with `docs/standards/document-style-guide.md`.
- [ ] PR template is fully completed.

## Commit Style

Use concise, imperative commit messages:

- `feat: add commissioning readiness checklist`
- `fix: clarify thermostat failover recovery steps`
- `docs: standardize maintenance runbook headings`

## Review Expectations

- At least one reviewer approval for non-trivial changes.
- Resolve comments before merge.
- Avoid direct pushes to `main` for substantive edits.
